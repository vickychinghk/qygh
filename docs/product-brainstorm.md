# 迷惑行为投稿协作网站 - 产品需求与头脑风暴草稿

创建日期：2026-05-11

## 目标

做一个给公众号编辑团队使用的网站，把飞书多维表格问卷收集到的“迷惑行为”投稿同步到我们自己的数据库里，方便编辑筛选、分组、排序、写吐槽语、投票确认，并最终导出为微信公众号文章 HTML。

用户不需要理解技术细节；技术方案、前后端实现、部署、数据库、API 对接和后续维护方案由 Codex 负责设计和实现。

## 当前已知业务流程

1. 在飞书新建多维表格。
2. 将多维表格开放成问卷，粉丝通过问卷投稿。
3. 投稿进入飞书多维表格，每条记录包含文字、图片和若干问卷字段。
4. 网站通过飞书 API 拉取多维表格记录。
5. 拉取后的记录缓存到我们自己的服务器数据库，避免重复请求飞书和重复下载图片。
6. 编辑登录网站后，对投稿进行筛选、归纳、排序、写吐槽语、点赞和最终勾选。
7. 按月份或期数整理成一篇篇公众号推送。
8. 一键导出微信公众号可用的 HTML。

## 飞书问卷/多维表格字段草案

字段名称以后以现有飞书多维表格为准，不反向设计字段模板。第一版先在本地完成账号系统和业务后台，等可以接入飞书鉴权后，再读取真实字段并映射。

- 投稿编号或飞书唯一记录 ID
- 上传图片
- 上传人的一句吐槽语
- 投稿人认为该迷惑行为所属学校
- 是否已征得同意
- 投稿时间
- 其他补充说明
- 审核状态
- 同步状态

## 网站核心功能草案

### 0. 登录与用户识别

- 第一版先做账号密码登录。
- 后续再接微信登录。
- UI 上需要用头像快速区分编辑。
- 第一版头像可以用用户名首字母或首字生成。
- 编辑人数预计约 30 人。

### 1. 飞书同步

- 手动点击刷新，从飞书多维表格拉取新增或更新记录。
- 可考虑定时自动同步，但需要控制频率。
- 使用飞书记录唯一 ID 做去重。
- 图片下载到服务器后压缩和缓存。
- 已同步过的图片不重复下载。
- 同步失败需要记录错误，方便重试。
- 投稿图片每条约 1-5 张，常见 1-2 张。
- 图片预估平均 1-2MB，真实大小待接入多维表格后统计。

### 2. 投稿库

- 查看所有已同步投稿。
- 按时间段筛选。
- 按学校、是否同意、审核状态等字段筛选。
- 支持手动添加到某一期公众号推送。
- 支持从某一期手动排除。

### 3. 推送/期数管理

- 每个月可以有多篇推送。
- 每篇推送内包含若干条迷惑行为投稿。
- 支持批量加入：例如选择某个时间段内的投稿。
- 支持手动增删投稿。
- 支持篇内排序：拖拽排序，必要时保留上移/下移按钮。

### 4. 编辑协作

- 多用户注册登录。
- 第一版先账号密码登录，后续再接微信登录。
- 每个编辑可以对投稿点赞或打 call。
- 每个编辑可以给每条投稿写无限条吐槽语。
- 吐槽语也可以被其他编辑点赞或打 call。
- 每条投稿最终只能采用一条吐槽语。
- 所有人都可以看到点赞、打 call 和最终勾选状态。
- 点赞用小星星，表示“我喜欢”。
- 打 call/确认用勾选，表示“确认采用”。
- 投稿被确认后，才正式进入该篇推送并在导出时出现。
- 吐槽语被确认后，才会在最终排版中出现。
- 所有编辑平等拥有确认权限。

### 5. 导出公众号文章

- 先做基础 HTML 导出。
- 后续再接入固定排版模板和微信公众号 HTML 规范。
- 导出内容应基于某一篇推送的已排序投稿和最终选中的吐槽语。

### 6. 使用端

- 仅编辑后台，不做公开粉丝页面。
- Mobile first，几乎完全按手机使用体验设计。
- UI 需要简练、扁平、高效。
- 尽量使用成熟组件库和 icon，不堆手写 UI。

## 初步技术方向候选

### 推荐方向：React / Next.js 全栈 Web 应用

- 前端：React + Next.js
- 后端：Next.js API Route / Server Actions，必要时再拆独立服务
- 数据库：PostgreSQL
- ORM：Prisma
- UI 组件：shadcn/ui + lucide-react
- 图片存储：本地开发先存在项目可写目录，正式部署再看服务器条件决定本地对象目录或兼容 S3 的对象存储
- 图片处理：服务端压缩生成 WebP/JPEG 缩略图
- 登录：第一版账号密码，后续微信网页授权
- 部署：单服务器部署，使用 PM2/Docker/Nginx 之一

### 为什么初步倾向这个方向

- 编辑使用的是浏览器和微信内网页，Web 应用最自然。
- 投稿、吐槽、点赞、分组、排序都适合关系型数据库。
- 服务器带宽有限，图片需要本地压缩和缓存，不能每次从飞书原图加载。
- 后续导出 HTML、接公众号排版模板，也适合服务端统一生成。

## 数据实体草案

- User：编辑用户
- Submission：迷惑行为投稿
- SubmissionImage：投稿图片
- Issue：公众号推送篇目，例如 2026 年 5 月第 1 篇
- IssueItem：某篇推送里的投稿及排序
- Comment：编辑写的吐槽语
- Reaction：点赞 / 打 call
- Selection：最终采用标记
- SyncJob：飞书同步记录

## 需要特别注意的约束

- 服务器上传/下载速度和流量有限。
- 飞书 API 不能高频重复拉取。
- 飞书图片不能每次现取，应缓存和压缩。
- 同步逻辑要能断点重试。
- 微信登录需要确认使用公众号网页授权、开放平台，还是先做账号密码登录。
- 微信公众号 HTML 规范较严格，导出功能先做基础版本，后续再精修模板。
- 当前没有服务器、域名、HTTPS、数据库环境。
- 第一阶段先本地跑起来，验证业务流程和 UI。

## 待确认问题

1. 每月大概多少条投稿？
2. 一篇公众号推送通常会选多少条投稿？
3. 飞书多维表格接入时，是用个人飞书应用、企业自建应用，还是你已有应用凭证？
4. 是否需要区分管理员和普通编辑，还是第一版所有编辑权限一致？
5. 投稿确认是否允许多人同时勾选，但最终按“勾选数”决定；还是一个人勾选后就直接进入最终导出？
6. 吐槽语确认是否同上：任何人勾选一条就作为最终吐槽语，还是需要管理员最终确认？

## 下一步头脑风暴方向

先把项目拆成几个阶段：

1. MVP：登录、飞书同步、投稿库、期数管理、吐槽和投票、基础 HTML 导出。
2. 协作增强：更细的权限、操作历史、冲突提示、批量操作。
3. 排版增强：微信公众号固定模板、图片排版、复制到公众号后台的兼容优化。
4. 运维增强：自动同步、图片压缩队列、备份、监控、失败告警。
